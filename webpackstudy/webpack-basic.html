    # webpack 基础

    # 1. webpack 包含哪些内容?

    # 2. webpack 五个核心概念

    1. entry
    2. output
    3. loader
    4. plugins
    5. mode
    1. entry

    webpack 入口, 指示 webpack 以哪个入口文件为起点开始打包, 分析构建内部依赖图

    1. output

    webpack 输出, 指示 webpack 打包后的 bundles 输出到哪里去, 以及如何命名

    1. loader

    模块转换器，将非 js 模块转换为 webpack 可以识别的 js 模块

    1. plugins

    扩展插件，webpack 运行的各个阶段，都会广播出对应的事件，插件去监听对应的事件.

    1. mode

    指示 webpack 使用相应模式的配置

    包含 production 和 development

    production 模式下, webpack 会自动开启代码压缩功能

    # 3. webpack 开发环境基本配置

    • 处理 css scss less 等文件, 并将 css 抽取成单独文件 css 兼容性处理 给 css 添加前缀 压缩 css

    • 开启source-map, 优化代码调试

    # 4. webpack 生产环境基本配置

    • 处理 css scss less 等文件 并将 css 抽取成单独文件 css 兼容性处理 给 css 添加前缀 压缩 css
    • js esLint 语法检查 js 兼容性处理
    • 处理 html 文件中图片 html-loader 压缩 html 文件
    • 处理图片

    # 5. webpack打包过程

    1. 初始化参数：shell webpack.config.js
    2. 开始编译：初始化一个Compiler対象，加载所有的配置，开始执行编译
    3. 确定入口：根据entry中的配置，找出所有的入口文件
    4. 编译模块：从入口文件开始，调用所有的loader, 再去递归地找依赖
    5. 完成模块编译：得到毎个模快被编译后的最终内容以及他们之间的依赖关系.
    6. 输出资源： 根据得到的依赖关系，组装成一个个包含多个module的chunk
    7. 输出完成：根据配置，确定要输出的文件名以及文件路径.

    # 6. webpack 中 loader 和 plugin 的运行时机 ？

    • 在 Webpack 中，Loader 和 Plugin 的运行时机完全不同，它们分别作用于构建流程的不同阶段。
    • 理解它们的执行顺序和触发时机，可以帮助你更好地优化构建性能或解决一些棘手的构建问题。以下是详细的解析：

    # 1. Loader 的运行时机

    Loader 的核心作用：对模块的源代码进行转换（如编译、压缩、转译等）。
    运行阶段：在 Webpack 的 模块解析（Module Resolution）和模块构建（Module Building）阶段 执行，早于打包结果生成。

    # 具体时机：

    1. 模块解析阶段：

      • 当 Webpack 遇到 import/require 语句时，会根据文件后缀名匹配对应的 Loader（通过 module.rules 配置）。
      • 例如：.js 文件会经过 babel-loader，.scss 文件会经过 sass-loader → css-loader → style-loader。

    2. 模块转换阶段：

      • Loader 按从右到左（或从下到上）的顺序链式调用（pitch 阶段则相反）。
      • 每个 Loader 接收上一个 Loader 的处理结果，最终输出 JavaScript 模块（或可被 Webpack 处理的格式）。

    # 示例流程（以 SCSS 文件为例）：

    // webpack.config.js
module: {
  rules: [
    {
      test: /\.scss$/,
      use: ['style-loader', 'css-loader', 'sass-loader'], // 执行顺序：sass-loader → css-loader → style-loader
    },
  ],
},
    1. sass-loader：将 SCSS 编译为 CSS。
    2. css-loader：解析 CSS 中的 @import 和 url()。
    3. style-loader：将 CSS 注入到 DOM 中。

    # 2. Plugin 的运行时机

    Plugin 的核心作用：扩展 Webpack 的功能，介入整个构建流程的各个生命周期（从启动到输出）。
    运行阶段：在 Webpack 的 编译（Compilation）、优化（Optimization）、输出（Emit）等阶段 通过钩子（Hooks）介入。

    # 具体时机（关键生命周期钩子）：

    1. 初始化阶段：

      • environment：Webpack 环境准备完毕。
      • initialize：初始化插件时触发。

    2. 编译阶段：

      • compile：开始编译前触发。
      • compilation：创建编译对象后触发（可访问模块依赖图）。
      • make：分析模块依赖时触发（Loader 在此阶段执行）。

    3. 优化阶段：

      • optimize：优化依赖树时触发（如代码分割、Tree Shaking）。
      • optimize-modules/optimize-chunks：优化模块或代码块时触发。

    4. 输出阶段：

      • emit：生成资源到输出目录前触发（可修改输出内容）。
      • after-emit：资源输出到目录后触发。
      • done：构建完成时触发。

    # 示例流程（以 HtmlWebpackPlugin 为例）：

    1. 在 compilation 阶段收集所有入口和资源信息。
    2. 在 emit 阶段生成 HTML 文件并注入打包后的 JS/CSS 路径。
    3. 在 after-emit 阶段将 HTML 写入磁盘。

    # 3. Loader 和 Plugin 的协作流程

    通过一个构建流程的完整示例说明二者的协作时机：

    1. 启动 Webpack：

      • Plugin 的 apply 方法被调用（如 new MyPlugin()）。
      • 触发 environment 和 initialize 钩子。

    2. 解析入口文件：

      • 遇到 index.js，根据配置的 Loader 规则处理 .js 文件（如 babel-loader）。
      • Loader 对源码进行转译。

    3. 递归处理依赖：

      • 遇到 import './style.scss'，调用 sass-loader → css-loader → style-loader。
      • Plugin 可能通过 module-build 钩子监听模块构建过程。

    4. 优化阶段：

      • Plugin（如 TerserWebpackPlugin）在 optimize-chunk-assets 钩子中压缩代码。
      • SplitChunksPlugin 分割代码块。

    5. 输出阶段：

      • Plugin（如 HtmlWebpackPlugin）在 emit 钩子中生成 HTML。
      • CleanWebpackPlugin 在 before-emit 钩子中清理旧文件。

    # 4. 关键区别总结

    特性 Loader Plugin
    作用对象 单个模块（文件） 整个构建流程
    运行时机 模块解析和构建阶段 贯穿整个生命周期（通过钩子）
    功能 转译源码（如 JSX → JS） 扩展功能（如打包优化、资源管理）
    配置方式 module.rules 数组 plugins 数组（需 new 实例化）

    # 5. 高级技巧（面试加分项）

    1. Loader 的 pitch 阶段：

      • Loader 可以通过 pitch 方法阻断后续 Loader 执行（如 style-loader!css-loader!sass-loader 中，若 style-loader.pitch 返回结果，则跳过后续 Loader）。

    2. 自定义 Plugin 钩子：

      • 通过 tapable 库创建自定义钩子，允许其他插件介入你的插件逻辑。

    3. 性能优化：

      • 在 compilation 阶段缓存 Loader 处理结果（如 cache-loader）。
      • 在 optimize-chunk-assets 钩子中并行处理代码压缩。

    # 回答示例（面试场景）

    Loader 在 Webpack 的模块构建阶段运行，负责对单个文件进行转译（如 Babel 转译 JS），执行顺序是从右到左链式调用。而 Plugin 通过钩子介入整个构建生命周期，比如 HtmlWebpackPlugin 会在 emit 阶段生成 HTML。

    举个例子：一个 SCSS 文件会先被 sass-loader 编译成 CSS，再经过 css-loader 解析依赖，最后通过 style-loader 注入到 DOM。而 MiniCssExtractPlugin 则会在优化阶段将 CSS 提取为独立文件。”

    # 7. webpack splitChunks 有哪3种方式 ？

    1. 按缓存组分割, cacheGroups
    2. 按 minSize、maxSize 分割

    可以依据模块的大小来分割代码。minSize 用于设置生成的块的最小大小，maxSize 用于设置生成的块的最大大小。

    1. 按 chunks 类型分割

    chunks 选项可用来指定分割哪些类型的 chunks，它有 3 个可选值：'async'、'initial' 和 'all'。

    module.exports = {
  //...
  optimization: {
    splitChunks: {
      chunks: 'async' // 只分割异步加载的 chunks
    }
  }
};
    • 'async'：仅分割异步加载的 chunks，也就是通过动态导入（如 import()）加载的模块。
    • 'initial'：只分割同步加载的 chunks。
    • 'all'：分割所有类型的 chunks，包括同步和异步加载的。

    面试加分项

    1. 优先级问题：
      cacheGroups 中的 priority 字段决定规则优先级，防止规则冲突（如 vendors 和 common 同时匹配时，优先级高的先生效）。

    2. maxInitialRequests 限制：
      初始页面加载时并行请求的 chunk 数量默认限制为 30，超出后会合并代码块，需根据项目调整。

    3. 运行时分离：
      结合 runtimeChunk: 'single' 将 Webpack 运行时代码单独拆分，避免公共代码变化导致缓存失效：

      optimization: {
  runtimeChunk: 'single',
}

    # 对比总结

    方式 触发条件 适用场景 配置关键
    同步分割 同步导入（import） 多入口共享的公共模块、第三方依赖 chunks: 'all' + minChunks
    异步分割 动态导入（import()） 按需加载的组件或路由 chunks: 'async'（默认）
    强制拆分 手动指定 关键库独立缓存或特殊模块拆分 enforce: true 或魔法注释
