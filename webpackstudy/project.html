    项目通用封装

    # 项目通用封装

    # 1. 前端封装请求

    1. 封装通用请求方法的好处？

    • 主要是为了一些通用的、前端需要增加的一些通信的请求
    • 也是为了简化业务里面去写多个请求的一种方式

    2. 怎么做的？

    1. 新建一个全局的基准地址文件

    需要一个对象，进行包裹，最后导出这个对象，暴露给外部使用

    const cfg = {
  "webroot": "http://xxx.com", // 请求域名
  "resourceUrl": "http://bbb.com/resource", // 图片资源域名
  "uploadUrl": "http://ccc.com"
}
module.exports = cfg
    1. 新建一个专门用来封装请求的文件
    • 定义一个对象，用来包裹请求方法，方便后续导出使用
    • 包含url, 特殊的method, 特殊的header，data 等参数
    • 成功的回调函数
    1. 使用
    • import 引入请求
    • 调用时，传入需要的参数，写一个函数用于接收请求后返回的结果
    • 失败的话，怎么捕获错误

    # 1.1 实例场景

    • 创建一个axios实例，并在实例中设置请求超时时间timeout、重试请求次数retry、请求间隔retryDelay。
    • 为实例设置相应拦截器，在请求返回错误的时候对错误做处理。
    • 如果请求次数已经达到设置好的请求次数，不再发送请求，返回最终的错误信息；
    • 如果请求次数还没有达到设置好的请求次数，就在停留retryDelay（请求间隔）时间后
    import axios from 'axios';

// 创建带有重试机制的axios实例
function createAxiosInstance(config = {}) {
  const {
    timeout = 10000,       // 默认超时时间10秒
    retry = 3,             // 默认重试3次
    retryDelay = 1000,     // 默认重试间隔1秒
    ...axiosConfig
  } = config;
  
  // 创建实例
  const instance = axios.create({
    timeout,
    ...axiosConfig
  });
  
  // 请求拦截器
  instance.interceptors.request.use(
    config => {
      // 初始化重试计数器
      config.__retryCount = config.__retryCount || 0;
      return config;
    },
    error => {
      return Promise.reject(error);
    }
  );
  
  // 响应拦截器
  instance.interceptors.response.use(
    response => response,
    error => {
      const config = error.config;
      
      // 检查是否需要重试
      if (!config || !config.retry) {
        return Promise.reject(error);
      }
      
      // 检查是否达到最大重试次数
      config.__retryCount = config.__retryCount || 0;
      if (config.__retryCount >= config.retry) {
        return Promise.reject(error);
      }
      
      // 增加重试计数器
      config.__retryCount += 1;
      
      // 创建新的Promise来处理重试间隔
      const delay = new Promise(resolve => {
        setTimeout(() => {
          resolve();
        }, config.retryDelay || retryDelay);
      });
      
      // 返回Promise，在延迟后重试请求
      return delay.then(() => instance(config));
    }
  );
  
  // 设置默认重试参数
  instance.defaults.retry = retry;
  instance.defaults.retryDelay = retryDelay;
  
  return instance;
}

// 使用示例
const apiClient = createAxiosInstance({
  baseURL: 'https://api.example.com',
  timeout: 5000,      // 5秒超时
  retry: 2,           // 最多重试2次
  retryDelay: 2000     // 每次重试间隔2秒
});

// 发起请求
apiClient.get('/data')
  .then(response => {
    console.log('请求成功:', response.data);
  })
  .catch(error => {
    console.log('最终请求失败:', error.message);
  });

    # 2. 前端开发痛点

    1. 浏览器兼容性问题

    痛点：

    • 不同浏览器对CSS、JavaScript的实现差异

    • 老旧浏览器不支持现代API（如IE对ES6+的支持问题）

    • 移动端浏览器碎片化严重

    解决方案：

    • 使用Babel进行代码转译

    • 采用PostCSS和Autoprefixer处理CSS兼容

    • 引入polyfill填补API缺失

    • 使用Can I Use等工具检查兼容性

    • 建立浏览器支持矩阵，明确兼容范围

    1. 状态管理复杂度

    痛点：

    • 大型应用中状态管理混乱

    • 组件间状态共享困难

    • 状态变更难以追踪和调试

    解决方案：

    • 采用Redux、MobX等状态管理库

    • 使用React Context API处理简单状态共享

    • 实施严格的状态变更规范

    • 使用Redux DevTools等调试工具

    1. 性能优化

    痛点

    • 首屏加载时间过长

    • 大型单页应用资源体积过大

    • 频繁DOM操作导致页面卡顿

    解决方案：

    • 代码分割和懒加载

    • 图片和资源优化（WebP、CDN）

    • 虚拟列表优化长列表渲染

    • Web Workers处理复杂计算

    • 使用性能分析工具（Lighthouse）

    1. 前端安全风险

    痛点：

    • XSS（跨站脚本攻击）

    • CSRF（跨站请求伪造）

    • 敏感数据泄露风险

    解决方案：

    • 输入输出编码/转义

    • 实施CSP（内容安全策略）

    • 使用HttpOnly和Secure Cookie

    • 敏感操作二次验证

    • 定期安全审计

    1. 前后端协作问题 痛点：

    • API接口变更频繁

    • 接口文档维护不及时

    • 数据格式不一致

    解决方案：

    • 使用Swagger/OpenAPI规范接口文档

    • 建立Mock Server进行并行开发

    • 实施TypeScript类型共享

    • 定期进行接口评审

    1. 开发体验问题 痛点：

    • 项目配置复杂（Webpack等）

    • 构建时间过长

    • 开发环境与生产环境不一致

    解决方案：

    • 使用Vite等现代构建工具

    • 优化构建配置（缓存、并行处理）

    • 容器化开发环境

    • 完善的HMR（热模块替换）

    • 代码分割好处
    1. 减少初始加载资源体积 核心优势：

    • 只加载渲染当前页面所需的代码（主包+当前路由的chunk）

    • 避免一次性加载整个应用的JavaScript和CSS

    • 典型场景：将路由组件、大型第三方库、非关键功能拆分为独立chunk

    1. 优化资源加载顺序
    // Vue中的路由懒加载
const Home = () => import('./views/Home.vue')
const About = () => import('./views/About.vue')

    优势：

    • 优先加载关键资源（当前视图必需代码）

    • 延迟加载非关键资源（如模态框、工具提示等交互组件）

    • 浏览器可以更早开始解析和执行关键代码

    1. 并行加载能力提升

    工作机制：

    • 主包加载同时，浏览器可以并行请求其他chunk

    • 利用HTTP/2的多路复用优势更高效

    1. 缓存策略优化 长期效益：

    • 修改业务代码不会使vendor包（第三方库）缓存失效

    • 用户再次访问时只需下载变更的chunk

    • 静态资源可以设置更长缓存周期

    1. 按需加载非首屏内容 典型应用场景：
    // 图片懒加载
<img v-lazy="'/path/to/image.jpg'" alt="description">

// 组件懒加载
const Chart = defineAsyncComponent(() => import('./components/Chart.vue'))

    优势：

    • 延迟加载首屏外的图片/组件

    • 优先完成可视区域(viewport)内容渲染

    • 节省带宽和计算资源

    实现建议

    1. 路由级分割：Vue Router/React Router天然支持

    2. 组件级分割：对大型组件使用defineAsyncComponent（Vue）或React.lazy

    3. 第三方库分割：将不常变的依赖单独打包

    复制
// webpack配置
optimization: {
  splitChunks: {
    chunks: 'all'
  }
}
    1. 预加载策略：对即将访问的资源使用<link rel="preload">

    注意事项

    • 过度分割会导致请求过多（平衡chunk数量和大小）

    • 需要配合Loading状态提升用户体验

    • 服务端渲染(SSR)需要特殊处理懒加载组件

    • 移动端弱网环境收益更明显
